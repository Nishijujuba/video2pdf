# Use kernel-issued task envelopes for every subagent

The workflow requires several specialized subagents with different inputs, output paths, schemas, and write permissions. Repeating those details in prompts causes path drift, stale dependency use, freeform report schemas, and unnecessary token consumption. Acceptance already demonstrates the value of a generated skeleton and an explicit artifact boundary.

## Considered Options

- Keep role contracts entirely in skill prompts: rejected because prompts cannot prove current paths, fingerprints, or schema versions.
- Define a different handoff format for every role: rejected because orchestration, validation, and recovery would remain fragmented.
- Give every subagent one kernel-issued task envelope with role-specific payloads: selected because common mechanics become deterministic while semantic work remains specialized.

## Decision

Run Initialization creates `workflow/tasks/` as part of the fixed Video Output Scaffold. Before spawning a Run-scoped Source Acquisition, Outline, Writer, Figure, Consistency, Independent Review, or repair subagent, the coordinator calls `task-prepare`. The Kernel creates a unique `workflow/tasks/<task-id>/task.json` and any required schema or skeleton files. Final Acceptance follows ADR 0056: `acceptance-prepare` creates its module-scoped task files under `review/acceptance/executions/<execution-id>/tasks/` for both Kernel and Legacy input bindings.

Every Subagent Task Envelope contains a discriminated Task Authority Binding, task identity, role, prerequisite state and input fingerprints, exact allowed read and write paths, required outputs, applicable schemas or skeletons, and the bounded semantic fields the agent may supply. A `kernel_run` binding includes `run_id`, expected `run_revision`, and the target Workflow Checkpoint. An `acceptance_execution` binding includes the execution identity, expected context revision, and immutable Kernel Final Evidence or Legacy Acceptance Input Set fingerprint. The envelope is immutable after launch. A subagent cannot modify the envelope, either coordination record, another task directory, or any undeclared path.

Skill prompts describe semantic goals, role-specific judgment, and how to use the supplied task. They reference the envelope instead of restating generated paths, artifact lists, fingerprints, or report shapes. Writer and Figure tasks may produce non-JSON artifacts, but their exact section identities and output paths still come from the envelope.

After a subagent returns, the coordinator calls the Task Completion Gate. `task-complete` validates the authority binding, path boundaries, required-file presence, schemas or file-format contracts, prerequisite freshness, and output fingerprints. Passing moves the Attempt to `validated_waiting_for_promotion`; only a later committed promotion may update the applicable authority record or Workflow Checkpoint. Subagent prose and self-declared success cannot advance workflow state.

An Acceptance Reviewer envelope may only narrow the existing allowed-artifact manifest and project acceptance contract. Its Acceptance Execution Context supplies task identity and transaction authority without granting additional content access or creating a Legacy Run Record. Pyramid reports remain generated by the evaluator wrapper; their task envelopes constrain inputs and register results without turning the agent into the report-schema owner.

## Consequences

All subagent launches become inspectable, resumable, and mechanically verifiable through one protocol. Prompt text becomes smaller and less path-sensitive. Role-specific schemas remain possible behind one common lifecycle. Task claiming, concurrent duplicate prevention, attempt history, cancellation, and stale-task recovery require follow-up decisions.
